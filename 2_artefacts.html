Initial post, other peer responses, summary post

I used the four main pillars (continuous and timely updates, filtering of traffic, intrusion detection and encryption) of network security as proposed by Ross (2008) to ensure I had a solid initial post. Many fellow students added good examples supporting the four pillars, but one student mentioned training, which isn’t covered by the model. I therefore added a 5th pillar to the model of Ross (2008) for me personally.
Also reading other post made me realize, that my view was very focused on modern day devices, whilst the reality is, that a network can and even sometimes must host older devices. I then realized that a hardened network is not just one additional layer of defense, it might well be the only layer of defense. This observation influenced my summary post, too.

References:
Ross, A. (2008) Security Engineering: A Guide to Building Dependable Distributed Systems. Second Edition. Indianapolis: Wiley Publishing Inc.

#3 NISM student website manual v2

I have set up many websites coding them myself using http & JavaScript or simply uploading WordPress or Joomla to a host. I have never used Amazon Web Service (AWS) before, so I was keen on using it, since knowledge around AWS is beneficial for my professional career.
The complexity of the AWS backend however is quite overwhelming, so the set up guide provided was key to get it running. I had several minor issues, where my tutor, my peers or Google could help me out.
The instructions to set up the website nonetheless had the following security faux paxs:
  • http instead of https
  • open SSH port, which isn’t strictly necessary, since server configuration can be done via AWS backend.
  • Usage of a deprecated PHP version.
  • Usage of an app, using outdated 3rd party libraires (e.g. Bootstrap)

#6 Seminar 2 preparation - TCP/IP v ISO/OSI

Team Nebula - Seminar 2 Presentation

After reading the article and prior seminar I had many arguments showing that ISO/OSI would have been worse than TCP/IP. My main point of the argument prior seminar was that many tools (PHP, MySQL, Apache, Firefox, …) could only be coded, since the TCP/IP standard had working implementations at the core, the open-source community could easily build on.
The seminar showed that my main conclusion was correct. TCP/IP was superior to ISO/OSI. Doug, our tutor added on my main argument by stating that ISO/OSI planned to sell costly licenses, which would have been likely to hinder a lot of good open-source software to be created. The internet as we know it now, would be different, if ISO/OSI would have prevailed.
